# Rehabilitation Assistant — мобильное приложение

React Native-приложение для сопровождения физической реабилитации. Помогает пациентам вести график тренировок, отслеживать эмоциональное состояние и получать доступ к библиотеке упражнений — всё в рамках структурированной программы восстановления.

В активной разработке, приближается к MVP.

## Обзор

У пациентов на реабилитации часто нет структурированных инструментов для управления восстановлением вне клиники. Приложение закрывает этот разрыв: пациент носит с собой программу реабилитации, график тренировок и библиотеку упражнений на телефоне, с профилем, адаптированным под его конкретные потребности восстановления.

## Моя роль

- **UI/UX-дизайн** — перевёл продуктовую идею в полноценный дизайн в Figma и интерактивный прототип, который команда использовала как референс при разработке
- **Экран профиля** — профиль пациента с полями данных, специфичными для реабилитации
- **График тренировок** — экран расписания с конструктором тренировок (создание, редактирование и управление личными планами тренировок)
- **Экраны информации и поддержки** — информация о приложении, FAQ и заготовка чат-интерфейса для будущей интеграции со специалистом или AI-консультантом
- **Рефакторинг фронтенда** — в середине разработки выявил несогласованность в структуре компонентов и экранов, инициировал и провёл рефакторинг для приведения кодовой базы к единой архитектуре
- **Архитектура бэкенда** — спроектировал серверную структуру с использованием AI-инструментов; Node.js + Express + PostgreSQL, впоследствии эволюционировавшая в микросервисную архитектуру (см. ниже)

## Стек технологий

| Слой | Стек |
|---|---|
| Мобильный клиент | React Native (JavaScript) |
| Бэкенд | Node.js + Express, микросервисная архитектура |
| База данных | PostgreSQL |

### Архитектура бэкенда

По мере роста продукта бэкенд был выделен из единого монолита в отдельные сервисы:

- **auth-service** — авторизация и аутентификация, выдача и проверка токенов
- **billing-service** — платежи и подписки
- **backend-service** — основная бизнес-логика (программы реабилитации, расписания, профили, контент упражнений)

Каждый сервис отвечает за свою зону, разрабатывается, деплоится и масштабируется независимо.

## Ключевые фичи, которые я реализовал

- **Конструктор тренировок** — пользователи могут составлять собственные тренировки и планировать их наряду с клиническими сессиями
- **Заготовка чата** — UI и структура маршрутизации для будущего чата со специалистом или AI-консультантом, спроектированная для подключения к любому бэкенду
- **Экран профиля** — хранит и отображает данные пациента, релевантные реабилитации, структурирован с расчётом на расширяемость по мере роста продукта

## Задачи и решения

**Задача:** функция расписания тренировок должна была работать с двумя типами сессий — клиническими (фиксированными, только для чтения) и созданными пользователем (редактируемыми) — так, чтобы UI не ощущался как два разных продукта.

**Решение:** оба типа сессий смоделированы через общий интерфейс расписания, а флаги владения определяют права на редактирование. С точки зрения пользователя это единый календарь; под капотом каждая сессия «знает», можно ли её изменять.

**Задача:** отсутствие опыта работы с Node.js, Express и проектированием реляционных баз данных. Нужно было спроектировать бэкенд с нуля — структуру API, аутентификацию и схему PostgreSQL, соответствующую модели данных приложения — одновременно разрабатывая экраны фронтенда.

**Решение:** использовал AI-инструменты для ускорения фазы проектирования бэкенда: сгенерировал первоначальную схему и структуру маршрутов, затем итеративно дорабатывал их, проверяя, задавая вопросы и адаптируя результат под реальные требования продукта. Такой подход сократил кривую обучения настолько, что удалось выпустить рабочий бэкенд, не блокируя остальную команду. По мере роста продукта этот бэкенд был далее разделён на микросервисы auth-service, billing-service и backend-service, описанные выше.

## Статус и результат

- Приближается к MVP — основные экраны готовы, идёт интеграция
- Бэкенд мигрирован с единого монолита на микросервисную архитектуру (auth-service, billing-service, backend-service)
- Спроектирован полный прототип в Figma, который служил единым источником истины для команды в процессе разработки, снижая неопределённость и переделки
